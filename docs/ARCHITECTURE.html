Lanmo 架构设计文档

版本 1.1  ·  设计定稿 · Phase 1 已完成，Phase 2 主干（Hybrid RAG + citation）已实现  ·  更新日期：2026-07-09

可扩展多能力 Agent 平台 · Poetry + FastAPI · LangGraph · Agent Skills · PostgreSQL + Neo4j

1. 项目定位

lanmo 是一个可扩展的多能力 Agent 平台（当前阶段以学习为主）。

维度说明
API 层Poetry + FastAPI
编排引擎LangGraph 状态机式工作流
专业能力Agent Skills 标准（agentskills.io），目录式可插拔
第一业务域写小说；后续扩展视频制作、导演、分镜等
架构思想Anthropic（Workflow/Agent、Context Engineering、Skills）+ OpenAI 工程纪律（增量构建、Guardrails、评测驱动）

能力分工：创作 vs 查证（读架构前先建立这个模型）

不是「每个请求都：RAG → Web Search → 模型整理」。
RAG / Web 是按需外挂；写小说、润色、续写等默认主路径靠 Skill + 多轮历史 + 模型，多数轮次不检索。
用户诉求负责模块典型场景
怎么写（角色、文风、格式）Agent Skills资深小说作家、导演、分镜师人设与输出约束
刚才聊了什么（短期记忆）PG 多轮历史 + compact_history续写上一段、改语气、多轮改稿
作品里已写了什么（长期原文/设定）RAG（PG + Neo4j，project 路由）「张三师父是谁」「第三章结尾原文」
库外/实时事实Web Search（realtime 路由或 Grader 兜底）「今天新闻」、查外部 API 说明
帮用户做事（存稿、导出）MCP（Phase 3）写文件、调绘图/ASR 等外部工具
生成与润色LLM generate在 Skill + 历史 +（可选）检索证据上创作

三条能力通道（可组合，非互斥三选一）

通道开关何时为 true写小说占比
C · 创作needs_creative续写、润色、改稿、闲聊；几乎每轮为 true最高
A · 作品库needs_project问库内设定/章节/人物；可与 C 同轮为 true（A+C）中
B · 外网needs_web时效/库外事实；可与 A 并行（A+B+C 少见）低

多轮对话（架构支持情况）

支持。每次 POST 是独立一次 LangGraph 运行，通过 conversation_id 串成同一会话：上一轮 save_messages 写入 PG → 下一轮 load_history 读回 → 注入 generate。详见第 14 章。

  • 第一版 API 范围：仅 chat 一个接口（POST SSE 流式问答已落地）
  • 数据喂养：明确不做 HTTP 接口；Ingestion 仅预留内部流水线与对接 Stub，Phase 2 用内部脚本/CLI灌测试数据，未来由既有喂养系统调用 Stub 写入 PG → Neo4j

3. 总体架构

单次 Chat 请求自 Client 至 PG 持久化的完整流程如下。步骤编号 ①–⑲ 与流程图连线标签及第 15 章数据契约一致。

请先建立正确心智模型（与「搜索型 Copilot」不同）：
① 每次请求先加载 Skill 与多轮历史（⑥ 读 PG）；
② route_question（⑧）输出能力开关 needs_project / needs_web / needs_creative，可组合（常见 A+C）；
③ 纯创作时仅 needs_creative，不检索、不 Web → compact_history → generate；
④ RAG 查的是你的作品库，不是「帮模型记住上一句」；记忆靠 PG 会话历史。

完整请求流程说明

以下步骤编号与流程图连线标签完全一致（①–⑲）。

④ 与 ⑤ 不是读两次 Skill。LangGraph 按节点顺序执行：④ 在 load_skill 节点内唯一一次访问 Skill Registry；⑤ 仅表示该节点执行完后，沿边进入下一个节点 load_history，不再访问 Skill。

先 Skill 后历史？没有 Skill 也读历史吗？

是。当前 LangGraph 主链为固定顺序、无条件分支：

load_skill → load_history → route_question → …
问题结论
先做什么？先执行 load_skill（匹配并注入 Skill prompt），再执行 load_history（读 PG 会话历史）
没匹配到 Skill 还读历史吗？仍然读。两节点之间没有「有 Skill 才继续」的条件边；load_skill 结束后总是进入 load_history
为什么？Skill 决定「怎么答」（角色/策略）；历史决定「之前说了什么」（多轮上下文）。二者独立，聊天都需要历史
没匹配到 Skill 怎么办？load_skill 内走兜底：使用配置中的默认 Skill（如 general），或仅保留基础 system prompt，不阻断后续流程

4.4 单向同步（非双向主从）

「双向主从」指 PG 与 Neo4j 互相当主库、两边都写、互相同步——复杂且易不一致，本项目明确不做。

原则说明
PG 是唯一真相源原始文档、业务数据只以 PG 为准
Neo4j 是派生索引从 PG 构建，可全量重建
同步方向PG → Neo4j（单向）
第一版chat 阶段可暂不跑同步

Neo4j 挂了？从 PG 重跑 Ingestion 重建即可。

4.5 数据存入与切割关系

不是「原始数据放 PG、只有关系放 Neo4j」，而是：
PG 存原始与业务真相；Neo4j 存切割后的可检索知识图谱（含 chunk 文本与向量）。
类型存储位置说明
结构关系（NEXT、HAS_CHUNK）Neo4j 边文档顺序
向量相似（语义近邻）Neo4j vector index查询时 kNN，不预存 SIMILAR_TO 边
知识关系（MENTIONS、RELATED_TO）Neo4j 边GraphRAG 多跳推理

4.6 Neo4j「节点」与「边」是什么？

Neo4j 是图数据库。图由两类基本元素构成：

概念英文含义类比
节点Node一个「东西」，带标签和属性地图上的「地点」
边Relationship / Edge两个节点之间的有向连接，带类型和属性地图上的「箭头/道路」

写法 (:A)-[:REL_TYPE]->(:B) 表示：从节点 A 沿类型为 REL_TYPE 的边，指向节点 B。

小说场景举例

// 节点：人物
(:Entity {name: "张三", type: "主角"})
// 节点：片段
(:Chunk {text: "张三拜李四为师...", index: 12})
// 边：MENTIONS — Chunk 提到了 Entity
(:Chunk)-[:MENTIONS]->(:Entity)
// 边：RELATED_TO — 人物之间的关系
(:Entity {name:"张三"})-[:RELATED_TO {relation:"师徒"}]->(:Entity {name:"李四"})
// 边：NEXT — 相邻章节片段的顺序
(:Chunk {index:12})-[:NEXT]->(:Chunk {index:13})

本项目中的三类「边」

边类型连接用途
HAS_DOCUMENT / HAS_CHUNK项目→文档→片段结构归属：这段文字属于哪本书哪一章
NEXTChunk→Chunk阅读顺序：检索时可向前后扩展上下文
MENTIONS / RELATED_TO / APPEARS_INChunk↔Entity↔Chapter知识关系：谁出场、谁和谁有关、在哪一章
边 ≠ 向量相似度。「张三和李四是师徒」存为 RELATED_TO 边（显式关系）；「这两段文字意思相近」靠 vector index 在查询时算相似度，一般不存 SIMILAR_TO 边，避免图爆炸。

4.7 Chunk ID 对齐（Hybrid RRF 前提）

Ingestion 写入时，PG 与 Neo4j 必须使用同一套 chunk_id（UUID 或 {document_id}:{index}），RRF 融合与 citation 回指均依赖此 id。

7.2 route_question 如何判断：路由级联（Routing Cascade）

业内成熟做法（2026 生产共识）不是单靠关键词，而是分层级联：便宜的层先判，置信度不够再升级到贵的层，把 LLM 调用压到最少流量。

层级做法成本/延迟本项目状态
L1 规则层关键词/正则命中明确意图（如「搜索一下」「第 X 章」）~0ms，零成本已实现；关键词表可在 config/app.yaml route.keywords 覆盖，内置默认见 app/graph/route_rule_constants.py
L2 语义路由意图示例句 embedding + 余弦相似度（如 semantic-router 库），~100ms 不耗 LLM低待接入真实语义 embedding 后启用（当前 HashingEmbedder 无语义，不适用）
L3 LLM 结构化输出规则未命中/置信度低时，LLM 按 JSON Schema 输出 RouteDecision~500-800ms已实现：app/graph/llm_route_classifier.py，失败自动回退规则层结果

级联策略（app/graph/route_cascade.py）：规则层先判；仅当 route.mode: hybrid 且规则置信度 < llm_confidence_threshold 时才调 LLM；LLM 超时/解析失败一律回退规则结果，保证路由永不阻塞主流程。模式配置见 config/app.yaml route 段（sec-08）。

  1. 意图拆分：一条消息拆多个子查询，例如「张三 师父」仅用于检索，完整 user 原文仍进 generate；LLM 层可输出 sub_queries，规则层把整句作为唯一 sub_query
  2. 并行：needs_project 与 needs_web 同时为 true 时，Hybrid 与 Web 并行，RRF/merge 后注入 generate（当前实现为顺序执行）

当前实现状态：L1 + L3 级联已实现（route.mode 默认 hybrid）；L2 语义路由与并行 merge 为后续增强。

14. 多轮对话与历史保留

Agent 的多轮对话能力由 PostgreSQL 会话存储 + 每轮 LangGraph 流水线共同实现：当前结构完全支持，Phase 1 起即为验收项之一。

14.1 多轮对话机制

第 1 轮 POST（无 conversation_id）
  → 服务端生成 conversation_id，message_start 返回给客户端
  → load_history：新会话，history = []
  → … → generate → save_messages：写入 user + assistant 到 PG

第 2 轮 POST（带同一 conversation_id）
  → load_history：从 PG 读取第 1 轮及更早消息 → state.history
  → route_question：按本轮问题重新计算 needs_* 开关（可与第 1 轮不同，如仅 C → A+C）
  → generate：Skill + history +（可选）检索 → 回复
  → save_messages：追加本轮消息

第 N 轮 … history 过长 → compact_history 压缩摘要到 conversations.summary
环节节点/存储多轮职责
会话标识conversation_id（HTTP + PG）客户端续聊时原样回传；省略则新建会话
图 checkpointthread_id（默认 = conversation_id）LangGraph 中断恢复；第一版与 conversation 1:1
读历史load_history每轮必执行（在 route 之前），与是否走 RAG 无关
写历史save_messages每轮结束追加 user/assistant；下一轮可读
长期记忆conversations.summary超 token 时 compact；更早内容以摘要形式进入 prompt
作品原文RAG / Neo4j不是多轮记忆替代品；存已发布章节/设定，按问题检索
区分「会话记忆」与「作品库」：「你刚才让我把打斗改短一点」→ history；「第三章里张三第一次出场写了什么」→ RAG 查作品库。二者互补，不互相替代。

通过配置文件设置对话历史保留时间，支持 Context Engineering 分层策略。

15.6 SSE 流式模型

Client POST → Chat API
  → graph.astream(state, config={ configurable: { thread_id } })
  → 每个 chunk 更新 state.answer / citations
  → FastAPI 映射为 content_delta / citation 事件
  → 图结束 → save_messages → message_end

15.6 持久化与失败策略

  • message_id 在 message_start 预生成（UUID），save 时幂等 upsert
