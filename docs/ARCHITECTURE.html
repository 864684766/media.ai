Lanmo 架构设计文档

版本 1.0  ·  设计定稿 · Phase 1 待实现  ·  更新日期：2026-07-07

可扩展多能力 Agent 平台 · Poetry + FastAPI · LangGraph · Agent Skills · PostgreSQL + Neo4j

1. 项目定位

lanmo 是一个可扩展的多能力 Agent 平台（当前阶段以学习为主）。

维度说明
API 层Poetry + FastAPI
编排引擎LangGraph 状态机式工作流
专业能力Agent Skills 标准（agentskills.io），目录式可插拔
第一业务域写小说；后续扩展视频制作、导演、分镜等
架构思想Anthropic（Workflow/Agent、Context Engineering、Skills）+ OpenAI 工程纪律（增量构建、Guardrails、评测驱动）

当前代码状态

仅有 GET /api/v1/chat（app/api/chat.py）问答骨架，返回简单字符串。

  • 第一版 API 范围：仅 chat 一个接口（目标演进为 POST SSE 流式问答）
  • 数据喂养：暂不设计接口，未来接入既有喂养系统改造
  • 路由注册：app/api/router_loader.py 自动扫描，无需手动改 router.py

完整请求流程说明

以下步骤编号与 Vue Flow 连线标签完全一致（①–⑲）。

④ 与 ⑤ 不是读两次 Skill。LangGraph 按节点顺序执行：④ 在 load_skill 节点内唯一一次访问 Skill Registry；⑤ 仅表示该节点执行完后，沿边进入下一个节点 load_history，不再访问 Skill。

先 Skill 后历史？没有 Skill 也读历史吗？

是。当前 LangGraph 主链为固定顺序、无条件分支：

load_skill → load_history → route_question → …
问题结论
先做什么？先执行 load_skill（匹配并注入 Skill prompt），再执行 load_history（读 PG 会话历史）
没匹配到 Skill 还读历史吗？仍然读。两节点之间没有「有 Skill 才继续」的条件边；load_skill 结束后总是进入 load_history
为什么？Skill 决定「怎么答」（角色/策略）；历史决定「之前说了什么」（多轮上下文）。二者独立，聊天都需要历史
没匹配到 Skill 怎么办？load_skill 内走兜底：使用配置中的默认 Skill（如 general），或仅保留基础 system prompt，不阻断后续流程

4. 存储架构

4.1 职责分离

库角色存储内容不存储
PostgreSQLSource of Truth users, projects, conversations, messages, documents, ingestion_jobs 图遍历、向量近邻
Neo4j派生知识索引（可重建） Chunk, Entity, 结构边, 知识边, vector index 通用 CRUD、会话历史

4.2 PostgreSQL 核心表（草案）

users            — 用户
projects         — 项目（小说/视频等），metadata JSONB 存设定/分镜
conversations    — 会话，含 summary 压缩摘要
messages         — 消息（role, content, created_at）
documents        — 原始文档（source, version, project_id）
ingestion_jobs   — 摄取任务状态
app_config       — 运行时配置快照（可选）
  • 全文检索：documents / messages 上 tsvector 索引
  • 灵活字段：projects.metadata JSONB（设定、分镜等跨业务扩展）

4.3 Neo4j 图谱模型（草案）

(:Project {id})-[:HAS_DOCUMENT]->(:Document {id, title})
(:Document)-[:HAS_CHUNK]->(:Chunk {id, text, embedding, index})
(:Chunk)-[:NEXT]->(:Chunk)
(:Chunk)-[:MENTIONS]->(:Entity {name, type})
(:Entity)-[:RELATED_TO]->(:Entity)
(:Entity)-[:APPEARS_IN]->(:Chapter {number})

4.4 单向同步（非双向主从）

「双向主从」指 PG 与 Neo4j 互相当主库、两边都写、互相同步——复杂且易不一致，本项目明确不做。

原则说明
PG 是唯一真相源原始文档、业务数据只以 PG 为准
Neo4j 是派生索引从 PG 构建，可全量重建
同步方向PG → Neo4j（单向）
第一版chat 阶段可暂不跑同步

Neo4j 挂了？从 PG 重跑 Ingestion 重建即可。

4.5 数据存入与切割关系

不是「原始数据放 PG、只有关系放 Neo4j」，而是：
PG 存原始与业务真相；Neo4j 存切割后的可检索知识图谱（含 chunk 文本与向量）。
类型存储位置说明
结构关系（NEXT、HAS_CHUNK）Neo4j 边文档顺序
向量相似（语义近邻）Neo4j vector index查询时 kNN，不预存 SIMILAR_TO 边
知识关系（MENTIONS、RELATED_TO）Neo4j 边GraphRAG 多跳推理

4.6 Neo4j「节点」与「边」是什么？

Neo4j 是图数据库。图由两类基本元素构成：

概念英文含义类比
节点Node一个「东西」，带标签和属性地图上的「地点」
边Relationship / Edge两个节点之间的有向连接，带类型和属性地图上的「箭头/道路」

写法 (:A)-[:REL_TYPE]->(:B) 表示：从节点 A 沿类型为 REL_TYPE 的边，指向节点 B。

小说场景举例

// 节点：人物
(:Entity {name: "张三", type: "主角"})
// 节点：片段
(:Chunk {text: "张三拜李四为师...", index: 12})
// 边：MENTIONS — Chunk 提到了 Entity
(:Chunk)-[:MENTIONS]->(:Entity)
// 边：RELATED_TO — 人物之间的关系
(:Entity {name:"张三"})-[:RELATED_TO {relation:"师徒"}]->(:Entity {name:"李四"})
// 边：NEXT — 相邻章节片段的顺序
(:Chunk {index:12})-[:NEXT]->(:Chunk {index:13})

本项目中的三类「边」

边类型连接用途
HAS_DOCUMENT / HAS_CHUNK项目→文档→片段结构归属：这段文字属于哪本书哪一章
NEXTChunk→Chunk阅读顺序：检索时可向前后扩展上下文
MENTIONS / RELATED_TO / APPEARS_INChunk↔Entity↔Chapter知识关系：谁出场、谁和谁有关、在哪一章
边 ≠ 向量相似度。「张三和李四是师徒」存为 RELATED_TO 边（显式关系）；「这两段文字意思相近」靠 vector index 在查询时算相似度，一般不存 SIMILAR_TO 边，避免图爆炸。

5. Provider Registry

设计目标：不硬编码厂商列表，通过配置切换任意模型。

适配器分层

适配器类型覆盖厂商示例协议
OpenAICompatibleProvider智谱 GLM、DeepSeek、Kimi、本地 vLLMOpenAI Chat Completions 兼容
AnthropicCompatibleProviderClaude 系列Anthropic Messages API
CustomProvider不满足上述协议的厂商独立 SDK 封装，注册到 Registry

配置示例

model:
  provider: zhipu          # Registry 中的 provider id，非硬编码枚举
  model: glm-4-flash
  api_base: https://open.bigmodel.cn/api/paas/v4/
  # api_key 从 .env 读取

切换模型只改配置，不改代码：

model:
  provider: deepseek
  model: deepseek-chat
  api_base: https://api.deepseek.com/v1

Provider Registry 伪代码

class ProviderRegistry:
    def register(self, provider_id: str, factory: ProviderFactory) -> None: ...
    def get(self, provider_id: str) -> BaseLLMProvider: ...

def build_llm(config: ModelConfig) -> BaseChatModel:
    adapter = registry.get(config.provider)
    return adapter.create(model=config.model, api_base=config.api_base, ...)

新增厂商 = 新增一个 Adapter 文件 + registry.register(...)，不动 LangGraph 主流程。

6. RAG 与 BM25

6.1 「不自写底层检索」的含义

自己做用 LangChain/LangGraph
手写 HNSW 向量算法Neo4jVector / LangChain Retriever
手写 BM25PG tsvector + LangChain 封装
手写 RRF 融合LangChain EnsembleRetriever / 社区组件
手写 Agent 状态机LangGraph 编排检索→评分→生成流程
 
LangChain = 检索组件（Retriever、Document Loader、Reranker）
LangGraph = 编排（何时检索、何时 Web、何时拒答）
自写     = 仅业务胶水层，不造向量引擎

仍需自己做的（业务编排层）：Ingestion 流水线、Adaptive Router/Grader Prompt 与阈值、双库写入协调、SSE 与会话管理。

6.2 BM25 说明

BM25 是一种关键词相关性排序算法（基于 TF、IDF、文档长度归一化），擅长精确词匹配，不擅长语义改写。

BM25Meilisearch
本质一种排序算法/公式完整搜索引擎产品
关系关键词匹配这一路含索引、分词、纠错、API、UI
本项目PG tsvector / Neo4j 全文索引第一版不引入
BM25 不是 Meilisearch。 Hybrid 检索 = 向量 + 关键词（BM25 路）→ RRF → Rerank。

查询类型对比举例

查询BM25（关键词）向量（语义）
张三 九幽魔帝强可
主角是谁弱强
第三章强一般

7. 检索准确性工程

准确性不靠数据库选型，靠检索流水线 + 纠错回路 + 评测。

生产默认栈

Hybrid Retrieval（Neo4j vector + PG 全文）
  → Reciprocal Rank Fusion（RRF）
  → Cross-Encoder Rerank（Top50 → Top5）
  → Neo4j 图扩展（2 跳）
  → LLM 生成（带引用）

路由策略（⑧ 三选一，非「先检索再 Web」）

路由分支代码枚举路径
A 查自己库project进入 Hybrid 检索链；仅 Grader 无证据时才兜底 Web
B 问实时/库外realtime直接 Web，不先检索
C 闲聊/续写general跳过检索与 Web，直达生成
检索链内兜底—走了 A 且重试耗尽仍无证据 → Web 或拒答（strict profile）

准确性 ROI 排序

  1. Ingestion 质量（切分、元数据、版本）
  2. Hybrid + Rerank
  3. Query Rewrite
  4. Neo4j 图扩展
  5. CRAG / Self-RAG 纠错
  6. Web 条件兜底
  7. 生成护栏 + RAGAS 评测

    base = apply_env_secrets(base)           # .env

    base = clamp(base, L1_HARD_CAPS)         # L1 安全边界
    return RuntimeConfig(**base)

11. MCP（Model Context Protocol）

MCP 是开放标准协议，用于 Agent 以统一方式调用外部工具和数据源。第一版：mcp.enabled: false，仅预留配置与 Client 接口。

定位与分工

RAG/检索  = 「我库里已经有的知识」 → 被动召回（读）
MCP       = 「我现在要去外部做事」 → 主动调用（读/写/API）
Skills    = 「我以什么角色/策略做事」 → 行为约束
LangGraph = 「按什么流程编排」 → 状态机
模块职责比喻
RAG从已有知识库召回图书馆查资料
MCP调用外部工具/API打电话/发邮件/操作软件
Skills领域角色与策略资深作家的人设
Provider换 LLM 模型换大脑

小说创作域 MCP 场景

MCP 应用做什么典型 MCP Server
读写稿件文件读取/保存 .md、.docx 章节filesystem
版本管理查看修改历史、对比章节差异git MCP
设定表导出人物/世界观导出为表格自定义 MCP + PG 查询
发布到平台调用开放 API（若有）第三方 API MCP
插图生成根据场景描述调绘图 APIimage generation MCP

短视频 / 导演 / 分镜域 MCP 场景

MCP 应用做什么典型 MCP Server
素材库浏览列出视频/图片/音频素材filesystem / 对象存储 MCP
字幕/转写调用 ASR 生成字幕transcription MCP
分镜脚本导出生成标准分镜表文件filesystem + 模板 MCP
视频元数据读取时长、分辨率、编码ffprobe / media MCP
配音/TTS根据台词生成旁白TTS API MCP
封面/缩略图自动生成视频封面image MCP

产品设计原则

  1. 内部知识 → RAG（PG + Neo4j）
  2. 外部动作/实时数据 → MCP
  3. 领域策略 → Skills
  4. 第一版 chat 阶段：MCP 全部预留，先不做

集成方式（预留）

  • 库：langchain-mcp-adapters（MultiServerMCPClient）
  • 传输：stdio（本地）/ streamable_http（远程）

14. 对话历史保留策略

通过配置文件设置对话历史保留时间，支持 Context Engineering 分层策略。

策略配置行为
原始消息保留history.retention_days: 90超期消息归档或删除
上下文压缩Profile 推导超 token 阈值触发 compact_history
压缩摘要存 PG conversations.summary保留长期上下文，减少 token

推荐分层：近期 N 轮保留原文；较早对话压缩为摘要；关键设定/人物/世界观永久保存；工具返回大块内容短期保留（如 7 天）。
