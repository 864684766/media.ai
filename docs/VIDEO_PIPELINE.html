1.1 与市面成熟工具对标（2026-07 复核）

对标对象：管线型（LTX Studio、mStudio）与切片型（Runway Gen-4.5 References、Kling 3.0 多镜模式、 Veo 3.1 ingredients、Seedance 2.0 多模态输入、Sora 2）。业界共识做法与本设计差距如下， 差距项已并入后续章节与 V1–V8 排期：

业界共识做法原设计状态本次补全
关键帧先行：先用图像模型出每镜关键帧（便宜、可人工确认），再图生视频；视觉决策在图像层锁定，视频模型只管运动 缺失——直接 render_shot 文生视频，bible 参考图来源也未交代 §2 流程与 §4 子图加 render_keyframe 节点；shots 增 keyframe_uri；排期入 V3
音频轨：对白 TTS、BGM、音效、字幕、口型是成片标配（Veo 原生音频、LTX lip-sync） 缺失——仅 shot.sfx 一个文本字段，合成只拼视频 §3.4 音频数据模型 + §6 tts_dialogue 契约；新增阶段 V8
全局风格锁：style bible 段落逐镜注入 + 统一调色，防止跨镜色调/画风漂移 缺失——只有角色/场景/道具三类实体锁 video_projects 增 style_bible/画幅字段；QA 增风格检查项
人工审核（HITL）：分镜、关键帧、成片有确认/驳回流 薄弱——仅 QA 超限「等待人工」一句 状态机增 awaiting_review；API 增确认接口；前端属 V6
成本控制：视频按秒计费（约 $0.03–0.17/秒），需成本记录与项目预算上限 缺失 shot_assets 增 cost；配置 video.budget（V7）
Provider 能力差异：参考输入数量（Sora 1–2 图 / Veo 图+视频 / Seedance 9 图 3 视频 3 音频）、Kling 原生多镜连贯模式 未考虑——render_shot 契约按单参考图假设 §6 契约改多参考输入；V7 增 Provider 能力矩阵要求
多候选（takes）：同镜并行生成 2–4 候选选优 只有失败重试语义 shot_assets 增 take_no（V3 起可选）

2. 目标端到端流

用户需求
  → director 策划（场景级视听方案，可仍走 chat）
  → storyboard 结构化分镜（JSON + 入库，非仅 Markdown）
  → 绑定 / 补全 角色·场景·道具·风格圣经（bible + style）
  → [HITL] 分镜确认（可跳过，配置项）
  → 关键帧生成（render_keyframe：图像模型，注入 bible 参考图 + style）
  → [HITL] 关键帧确认（可跳过）
  → 按镜切片生成（render_shot：图生视频，注入关键帧 + 上一镜尾帧）
  → 一致性校验（continuity_check）
       ├─ 不通过 → 重生成该镜（禁止入合成队列）
       └─ 通过 → 下一镜 / 全部通过
  → 音频轨（tts_dialogue / BGM / 音效 / 字幕，可与渲染并行）
  → 时间轴合成（compose_timeline，视频 + 音频 + 字幕）
  → [HITL] 成片审片 → 成片落盘 + 元数据回写

实现时建议用 LangGraph 视频子图或异步 Job Worker；不得再假装「一步 LLM 回复 = 成片」。 关键帧先行是业界共识（见 §1.1）：构图/光照/人物姿态在便宜的图像层锁定并可人工确认， 视频模型只负责运动与时序，能大幅降低漂移与返工成本。

状态机（镜头级）

状态含义允许迁移
draft分镜已写入，未校验实体→ validated / rejected
validatedcharacter/scene/prop ID 齐全且存在于 bible→ keyframing（或直出 rendering，配置项）
keyframing关键帧生成中（图像模型）→ keyframe_ready / failed
keyframe_ready关键帧已出，视 HITL 配置等待确认→ rendering / awaiting_review
awaiting_review等待人工确认（关键帧或 QA 超限）→ rendering（通过/修改后）/ rejected
rendering切片生成中→ rendered / failed
rendered切片已落盘，待 QA→ qa_passed / qa_failed
qa_failed一致性未过→ rendering（重试，限次）或 awaiting_review
qa_passed可入合成时间轴→ composed（随 Job）
failed生成硬失败→ keyframing / rendering 或终止

3. 数据模型（设计稿）

复用现有 project_id 命名空间；表名与字段为设计约定，实现时落 PG（真相源），大文件走对象存储/本地目录。

3.1 项目与圣经

实体关键字段说明
video_projects project_id、title、description、target_duration_sec、aspect_ratio、resolution、fps、style_bible、budget_limit_usd、status 可与业务 project 一对一。description 为选填项目说明（≤500 字，前端 ProjectSummaryCard 展示）；style_bible 为全局风格段落（画风/色调/负面词），逐镜逐帧原文注入防跨镜风格漂移；画幅/分辨率/帧率为项目级约束，所有镜头继承
character_bible character_id、name、appearance、costume、age_band、ref_image_urls[]、voice_id?、lock_version 角色外观锁定；生成前强制注入。ref_image_urls 可由「角色定妆」流程生成（图像模型 + 人工挑选，V2 交付物）；voice_id 供 V8 TTS 音色一致
scene_lock scene_id、name、setting、lighting、ref_image_urls[] 场景布景/光照锁定
prop_inventory prop_id、name、description、ref_image_urls[] 跨镜道具不得消失/换样

3.2 分镜与资产

实体关键字段说明
shots shot_id、shot_no、duration_sec、shot_size、camera、action、dialogue、character_ids[]、scene_id、prop_ids[]、keyframe_uri、transition、status 机器可读分镜；状态见上表。keyframe_uri 为已确认关键帧；transition 为与下一镜的转场（cut/dissolve 等），场景跳变时必填
shot_assets shot_id、asset_type(keyframe/clip)、clip_uri、last_frame_uri、provider、gen_params、attempt、take_no、selected、cost 关键帧/切片与尾帧。take_no 支持同镜多候选选优（业界常见 2–4 takes）；selected 标记入选；cost 记录该次生成费用（视频按秒计费，须可核算）
compose_jobs job_id、project_id、timeline、audio_tracks、subtitle_uri、output_uri、status 仅纳入 qa_passed 镜头；音轨与字幕见 3.4

3.4 音频与字幕（V8 · 成片标配）

市面成熟工具（Veo 原生音频、LTX lip-sync、mStudio 音频混轨）均把音频视为成片必备； 本设计将其独立为 V8 阶段，数据先行预留：

实体关键字段说明
audio_assets audio_id、shot_id?、track_type(dialogue/bgm/sfx)、uri、duration_sec、voice_id?、source(tts/library/upload) 对白经 tts_dialogue 生成并绑定 shot；BGM 可跨镜；音效对应 shot.sfx
字幕 compose_jobs.subtitle_uri（SRT/ASS） 由 shots.dialogue + 音频时长自动生成；合成时可烧录或外挂
角色声音 character_bible.voice_id 与外观同级的一致性资产：同一角色全片同一音色

  → storyboard_json   # LLM → Shot[]，写 shots(draft)
  → validate_entities # 校验 bible ID；缺则 rejected / 提示补全
  → review_gate       # HITL：分镜确认（video.review.storyboard 开关）
  → render_keyframes  # 图像模型出每镜关键帧（便宜、可并行）

  → audio_pipeline    # V8：tts_dialogue / BGM / 字幕（可与渲染并行）
  → compose           # compose_timeline（视频 + 音轨 + 字幕）
  → persist_output    # 成片 URI + 成本汇总 + Job 结束

5. 一致性严控（长视频必选）

长视频必须严控：人物形象、场景、动作、物品、行为跨镜一致，禁止张冠李戴。 QA 失败的镜头不得进入合成队列。

5.1 生成前（强制注入）

  1. 项目级 style_bible 段落逐字前置到每次图像/视频生成 prompt（业界防画风漂移的第一手段）
  2. 本镜 character_ids / scene_id / prop_ids 对应的 bible 全文 + 参考图 URL（同一角色全片复用同一组参考图，不得每镜换图）
  3. 本镜已确认 keyframe_uri 作为图生视频首帧锚定
  4. 上一镜 last_frame_uri（若存在）作为连续性锚定
  5. lock_version：bible 变更后未重新校验的镜头不得渲染

5.2 生成后（QA）

检查项V1 规则版后续增强
实体齐全ID 均存在于 bible—
场景跳变相邻镜 scene_id 变更须有 transition 标注视觉场景分类
人物一致同 character_id 不得换名/换装描述冲突人脸/服装嵌入比对
道具连续剧情未交代不得消失目标检测 Stub
风格/色调gen_params 中 style_bible 注入完整（规则可查）相邻镜色彩直方图/风格嵌入比对；合成期统一调色
技术规格切片分辨率/帧率/画幅与 video_projects 一致（ffprobe）—
内容安全Provider 拒绝（安全策略）视为 failed 并记录原因，不盲目重试提示词自动改写建议

5.3 重试与成本策略

  • 单镜默认最多 N 次（配置项，如 video.qa.max_retries: 3）
  • 超限 → 镜头置 awaiting_review，Job 暂停，等待人工确认或修改分镜/bible
  • 可选多候选：同镜并行 video.render.takes（默认 1）个候选，QA 选优入 selected
  • 项目累计成本达 video_projects.budget_limit → Job 暂停并告警，防止 QA 循环烧钱（视频按秒计费）

6.1 Provider 能力矩阵（V7 选型约束）

render_shot 契约按参考输入列表设计（而非单参考图），因为各家能力差异大： Sora 2 / Kling 3.0 收 1–2 张参考图；Veo 3.1 收图 + 视频片段且带原生音频； Seedance 2.0 单次可收 9 图 + 3 视频 + 3 音频；Kling 另有原生多镜连贯模式（一次 Job 3–12 镜自动保持人物/光照）。 V7 实现适配器时须声明各 Provider 的 max_ref_images / max_ref_videos / native_audio / multi_shot 能力位（配置进 app.yaml video providers 段），编排层据此裁剪注入的参考输入或改走多镜模式。

7.4 V4 连续性 QA 契约（已实现 · 规则版）

POST qa：扫描 status=rendered 镜头，按镜号顺序执行规则检查（实体齐全、关键帧已落盘、场景跳变须 transition、同场景道具不得无故消失）。通过 → qa_passed；失败 → qa_failed（qa_attempts+1）；达 video.qa.max_retries → awaiting_review。

{
  "project_id": "demo",
  "passed_count": 1,
  "failed_count": 1,
  "awaiting_review_count": 0,
  "failures": [
    {
      "shot_id": "...",
      "shot_no": "2",
      "status": "qa_failed",
      "reasons": ["missing_transition: s1 -> s2"]
    }
  ]
}

CLI：poetry run python scripts/run_continuity_qa.py --project demo（须先 render）。失败镜头不得进入 V5 合成队列。

7.7 V7 Provider、成本与预算契约（已实现）

配置：video.providers.active 选择关键帧/切片/合成实现；video.providers.matrix 声明能力位与 cost_per_second；video.budget 控制 default_limit_usd 与熔断开关。本地 ffmpeg 模式见 FFMPEG_PATH（可选，默认识别 PATH）。

GET cost：/api/v1/video/projects/{project_id}/cost 返回累计 total_cost_usd、budget_limit_usd、remaining_usd、active_providers。

POST render / compose：启动前按配置校验预算，超限返回 409，detail.reason=budget_exceeded。切片 shot_assets.cost 按 duration_sec × cost_per_second 记账。

Provider 模式：active.clip=local_ffmpeg 且系统有 ffmpeg 时产出 clip.mp4；active.compose=local_ffmpeg 且全部 clip 为 mp4 时产出 timeline.mp4。无 ffmpeg 时 render/compose 返回 503（FfmpegRequiredError），不再 silent 回退 JSON Stub。

{
  "project_id": "demo",
  "total_cost_usd": 0.15,
  "budget_limit_usd": 100.0,
  "remaining_usd": 99.85,
  "budget_exceeded": false,
  "asset_count": 4,
  "active_providers": {
    "keyframe": "stub",
    "clip": "stub",
    "compose": "stub_json"
  }
}

7.8 V8 音频与字幕契约（V8.5 增强）

POST audio：扫描 project 下含 dialogue 的镜头，按 character_bible.voice_id 写出对白 WAV，登记 audio_assets；并生成 subtitles.srt 供合成引用。

TTS Provider：video.audio.tts_provider — stub（静音占位）或 edge_tts（Edge 在线语音，需 edge-tts 包与网络）。

POST compose（V8.5）：active.compose=local_ffmpeg 且 mux_audio=true 时，将 audio_tracks 对白轨按镜头时间轴混入 timeline.mp4；时间轴 JSON 仍含 subtitle_uri。

字幕烧录（G3）：video.audio.compose.burn_subtitles=true 且存在 subtitles.srt 时，合成后用 ffmpeg subtitles 滤镜烧录进 mp4；响应 subtitles_burned 供前端展示。

配置：video.audio.tts_provider、video.audio.edge_tts.default_voice、video.audio.edge_tts.voice_map；产物根目录 video.assets_dir。TTS 失败时降级 stub 静音并写 warning 日志。

7.8.1 V8.6 BGM 导入与混轨（已实现）

POST audio/bgm：上传 mp3/wav/m4a/aac 作为全片 BGM，写入 audio_assets（track_type=bgm，shot_id 为空）；覆盖旧 BGM。

GET audio/tracks：列出对白 / BGM / 音效登记；media-web BgmUploadSection 预览与管理。

DELETE audio/bgm/{audio_id}：删除 BGM 登记与磁盘文件。

compose 混轨：存在 BGM 且 video.audio.bgm.default_volume 配置时，在对白 mux_audio 之后用 ffmpeg amix 混入 BGM（音量可配，默认 0.25）。

配置：video.audio.bgm.default_volume（0–1）。SFX（track_type=sfx）表结构已预留，按 shot 绑定为后续项。
